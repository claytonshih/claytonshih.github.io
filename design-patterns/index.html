🧩 設計模式 Design Patterns

前人整理的「常見問題標準解法」——不是程式碼範本，而是解決問題的套路。

SOLID 五原則

設計模式背後的共同精神，可以濃縮成五條原則，合稱 SOLID。它們是「判斷設計好不好」的準繩。

字母原則白話
S單一職責一個類別只做一件事，改動的理由只有一個。
O開放封閉對「擴充」開放、對「修改」封閉——加功能不必動舊碼。
L里氏替換子類別要能無痛替換父類別，不破壞行為。
I介面隔離介面要小而精，別逼使用者依賴用不到的方法。
D依賴反轉依賴「抽象」而非「具體」，高層不該綁死低層細節。
看模式時可以想：「這個模式幫我守住了哪一條原則？」多數模式都是為了 O（開放封閉） 與 D（依賴反轉） 而生。

單例 Singleton 創建型

確保一個類別「全程只有一個實例」，並提供一個全域取用點。

比喻：一個國家只有一位總統。不管你從哪個部門去問「總統是誰」，拿到的都是同一個人。
什麼時候用：設定檔、資料庫連線池、日誌記錄器、快取——這些「應該全系統共用同一份」的東西。
class Config {
  constructor() {
    if (Config.instance) return Config.instance;
    this.data = {};
    Config.instance = this;
  }
}

互動：不管建幾次，都是同一個

⚠️ 小心濫用：Singleton 本質是「全域變數」，過度使用會讓模組互相隱形耦合、也不好做單元測試。

工廠 Factory 創建型

把「決定要 new 哪個類別」的邏輯集中到一個工廠函式，呼叫端只說「我要什麼」。

比喻：去飲料店你只說「我要珍奶」，店員（工廠）決定用哪些原料、怎麼做。你不用自己進廚房組裝。
什麼時候用：要產生的物件種類很多、或未來會新增種類時。呼叫端不必到處寫 new。

互動：跟工廠點一杯

工廠待命中…

建造者 Builder 創建型

分步驟組裝一個複雜物件，把「一長串參數」變成一句句清楚的鏈式呼叫。

比喻：Subway 點餐——選麵包、加料、選醬、要不要烤，一步步堆疊，最後 .build() 送出成品。
什麼時候用：物件有很多可選參數時，避免看不懂的超長建構子。

互動：自己堆一個漢堡

轉接器 Adapter 結構型

在兩個「介面對不上」的東西中間夾一層轉接，讓它們能一起運作。

比喻：出國用的萬用轉接頭。你的插頭沒變、插座沒變，中間那個轉接器把它們兜起來。
什麼時候用：要接第三方套件或舊系統，但它的方法名稱／格式跟你目前的程式不相容時。
// 舊 SDK 的方法叫 makePayment，但我們全站都用 pay()
class PayAdapter {
  constructor(oldSdk){ this.sdk = oldSdk; }
  pay(amount){
    return this.sdk.makePayment(amount, "TWD");
  }
}

裝飾者 Decorator 結構型

不改原本的類別，用「一層層包起來」的方式動態疊加功能。

比喻：一杯基底咖啡，可以加奶泡、加糖漿、加奶油——每加一層都還是「咖啡」，但功能不斷疊上去。
什麼時候用：想給物件加功能，但用繼承會爆炸（各種組合要開一堆子類別）時。裝飾者用組合取代繼承。

互動：一層層裝飾你的咖啡

外觀 Facade 結構型

在一堆複雜的子系統前面，放一個「簡單的統一入口」，讓外部只需呼叫一個方法。

比喻：看電影用的「一鍵觀影」按鈕——背後幫你關燈、放下投影幕、開播放器、調音響，但你只按一顆鈕。
什麼時候用：子系統很多、步驟很繁瑣，想給使用者一個乾淨簡單的介面時。

互動：一鍵啟動家庭劇院

觀察者 Observer 行為型

一個「主題」狀態改變時，自動通知所有「訂閱者」。一對多的自動廣播。

比喻：YouTube 訂閱。你按下訂閱（註冊為觀察者），頻道一發新片，所有訂閱者立刻收到通知。
什麼時候用：事件系統、UI 資料綁定、通知推播——「一處變動、多處要跟著反應」時。

互動：訂閱頻道並發片

目前訂閱者：（尚無）

策略 Strategy 行為型

把「可以互換的演算法」各自包成一個策略，執行時再決定要用哪一個——不必寫一堆 if/else。

比喻：導航 App 的「最快 / 最短 / 避開收費」。目的地不變，你隨時切換「怎麼走」的策略，路線就跟著換。
什麼時候用：同一件事有多種做法要動態切換時（不同運費計算、不同排序規則、不同折扣方案）。

互動：切換結帳策略（原價 $1000）

迭代器 Iterator 行為型

提供一致的方式「一個接一個」走訪集合裡的元素，不必知道它內部長怎樣。

比喻：電視遙控器的「下一台」。你不需要知道電視怎麼儲存頻道清單，按一下就跳到下一個。
什麼時候用：想統一走訪不同資料結構的方式。JS 的 for...of 就是靠它。

互動：手動點 next()

模式類別一句話典型場景
Singleton 單例創建全系統只留一個實例設定檔、連線池、日誌
Factory 工廠創建集中決定 new 哪個類別依型別產生物件
Builder 建造者創建分步組裝複雜物件多可選參數的建構
Adapter 轉接器結構讓不相容介面能對接包裝第三方／舊 SDK
Decorator 裝飾者結構一層層動態加功能咖啡加料、串流加密
Facade 外觀結構複雜子系統的簡單入口一鍵完成多步驟
Observer 觀察者行為狀態變動自動通知訂閱者事件、UI 綁定、推播
Strategy 策略行為可互換的演算法動態切換運費、折扣、排序規則
Iterator 迭代器行為統一走訪集合元素for...of、自訂集合
學習建議：入門先熟這四個最常用的——Singleton、Factory、Observer、Strategy。其餘的等實際遇到對應問題再回來查即可。模式是工具不是目標，程式讀起來清楚、好改，比「用了幾個模式」重要得多。
